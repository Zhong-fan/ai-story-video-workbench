# 小说与有声视频生成系统正式方案

更新时间：2026-05-19

## 一、文档目的

本文档用于明确本项目的产品目标、设计原则、核心流程、视频约束体系、重生成机制和工程实现方向。

本文档的基本判断是：

- 小说生成本身相对轻。
- 视频生成约束明显更重。
- 真正决定系统产品价值的，主要是视频侧的可控性、一致性、可修改性和可重生成能力。

因此，本文档将采用“小说轻约束、视频重约束”的设计思路。

## 二、项目目标

本项目的目标是构建一套可持续使用的生成系统，使用户和团队成员能够：

- 生成符合需求的小说内容。
- 基于小说内容生成视觉短片，并在需要时叠加对白、配音、字幕和音效。
- 在生成后对结果提出意见，并推动系统按意见进行局部或整体重生成。
- 在流程复杂、细节繁多的前提下，仍然保持产品使用上的简洁性。

本项目追求的不是“模型一次生成一个结果”，而是“系统持续逼近用户想要的结果”。

## 三、总体设计判断

### 1. 小说和视频不是同一难度问题

小说主要是文字生成问题，虽然也需要题材、文风、人物、节奏等控制，但本质上仍然是相对单一媒介的内容生成。

视频不同。视频天然包含更多维度：

- 人物外观。
- 人物年龄感。
- 服装一致性。
- 场景空间关系。
- 镜头景别。
- 机位和运镜。
- 光影与色彩。
- 情绪氛围。
- 配音（可选项，不作为默认必选链路）。
- 字幕。
- 配乐。
- 音效。
- 镜头衔接。
- 视频片段合成。

因此，本项目不应把小说和视频设计成同等复杂度。更合理的方式是：

- 小说侧保留必要约束。
- 视频侧投入主要设计精力。
- 工程能力优先服务视频链路。

### 2. 产品重点不在“能生成”，而在“能改到满意”

用户真正的需求通常不是“随便给我一个结果”，而是：

- 给我一个方向大体正确的结果。
- 我指出哪里不对。
- 系统继续改。
- 最终越来越接近我想要的东西。

所以本系统的核心竞争力应该来自：

- 约束体系。
- 锁定体系。
- 反馈体系。
- 重生成体系。
- 任务和版本管理体系。

### 3. 先做通用能力，不做过重特化

系统需要支持用户提出强参考方向，例如希望在风格、情绪、镜头语言、气质上高度接近某个明确作品方向。

但系统不应起步就为每一类作品做专用分支，也不应起步就把所有低频需求做成重型配置系统。

正确策略是：

- 先抽象共性的约束能力。
- 先服务高频用户需求。
- 当某类需求反复出现时，再沉淀为模板或增强能力。

## 四、产品定位

本项目是一个“创作约束驱动”的一体化生成系统。

它不是单一的写作工具，也不是单一的视频工具，而是一套从需求到内容、从内容到镜头、从镜头到音视频成片、再从反馈回到重生成的完整工作流系统。

主链路如下：

```text
用户需求
-> 项目设定
-> 小说概要
-> 小说正文
-> 正文定稿
-> 分镜生成
-> 视频约束补全
-> 视觉资产与镜头首帧准备
-> 视频片段生成
-> 可选对白、配音、字幕和音效
-> 合成最终视频
-> 用户反馈
-> 局部或整体重生成
```

## 五、设计原则

### 1. 简单优先

- 默认流程必须短。
- 默认入口必须少。
- 普通用户先走一键主路径。
- 高级控制项只在需要时展开。

### 2. 视频优先

- 小说生成保证可用即可，不做不必要的重设计。
- 主要设计资源投入视频侧。
- 角色、场景、镜头、音频、合成和重试能力优先建设。

### 3. 约束优先

- 没有约束，模型输出就不可控。
- 视频越复杂，越不能依赖一句大提示词。
- 必须把风格、角色、场景、镜头、声音、禁止项拆成结构化约束。

### 4. 用户反馈优先

- 用户不满意是常态，不是异常。
- 系统必须允许用户对每个关键结果提意见。
- 这些意见必须真正进入下一轮生成，而不是只做文本备注。

### 5. 锁定优先

- 用户已经确认的东西必须可锁定。
- 锁定后不能被后续任务轻易覆盖。
- 这是保证生成稳定性的核心条件。

当前实现上，这条原则已经开始落地到真实代码路径：

- 已锁定角色三视图不会被普通预检重生成覆盖。
- 已锁定对白脚本不会被普通对白脚本生成覆盖。
- 已锁定对白/旁白音频不会被普通配音生成覆盖。
- 已锁定镜头首帧不会被普通首帧生成覆盖。

### 6. 工程服务产品

- 异步任务不是为了复杂化，而是为了让长链路可执行。
- 版本管理不是为了完美，而是为了可回退和可比对。
- 可观察性不是为了开发方便，而是为了让用户看得懂系统在做什么。

## 六、用户需求模型

用户通常会有以下类型的需求：

- 我想写一个什么故事。
- 我想让它呈现出什么气质。
- 我希望人物是什么感觉。
- 我希望视频长什么样。
- 我希望声音是什么风格。
- 我不满意某一段、某个镜头或某个角色时，希望只改那一部分。

因此系统应支持三类输入：

### 1. 基础输入

- 作品名称。
- 类型与题材。
- 内容简介。
- 目标长度。

### 2. 风格输入

- 文风方向。
- 视觉方向。
- 情绪方向。
- 参考作品或参考风格。
- 禁止项。

### 3. 修订输入

- 哪里不满意。
- 希望保留什么。
- 希望加强什么。
- 希望删除什么。

## 七、小说侧设计

小说不是本项目最复杂的部分，因此设计原则应是“够用、可修订、不过重”。

### 1. 小说侧目标

小说侧只需要稳定支持：

- 概要生成。
- 用户反馈后修订概要。
- 锁定概要。
- 根据锁定概要生成正文。
- 对正文进行重写。
- 将章节定稿，作为后续分镜和视频的输入。

### 2. 小说侧约束

建议保留以下核心约束即可：

- 题材。
- 世界观。
- 角色。
- 角色关系。
- 情节主线。
- 章节节奏。
- 文风方向。
- 禁止项。

不必在起步阶段把小说侧做成过重的复杂规划系统。小说侧重点是：

- 给视频提供稳定的剧情输入。
- 给分镜提供清晰的场景和事件基础。
- 让用户能快速改正文，而不是陷入过多结构化编辑。

### 3. 小说侧应支持的修改能力

- 整体重写章节。
- 根据反馈修改章节。
- 后续逐步支持局部段落重写。

小说侧不应过度工程化，但必须具备可修改性。

## 八、视频侧设计总则

视频是本项目的核心难点，因此必须单独建立完整约束体系。

视频生成的基本问题不是“有没有模型”，而是：

- 结果是否稳定。
- 结果是否一致。
- 结果是否接近用户要求。
- 结果是否能被局部修改。
- 失败后是否能继续。

因此视频侧必须重点解决六件事：

1. 风格约束。
2. 人物一致性。
3. 场景一致性。
4. 镜头可控性。
5. 音频包装。
6. 局部重生成。

## 九、视频约束体系

### 1. 项目级风格约束

项目级风格约束是所有视频生成的上游控制源。

它至少应包含：

- 画面媒介。
- 总体气质。
- 色彩倾向。
- 光影倾向。
- 镜头语言倾向。
- 情绪方向。
- 参考风格。
- 禁止项。

例如用户提出强参考方向时，系统不应只保存一句“像某作品”，而应拆成结构化风格块，供后续所有生成流程继承。

### 2. 角色约束

角色约束应至少包含：

- 角色名称。
- 年龄感。
- 外观描述。
- 发型与服装。
- 气质。
- 情绪基调。
- 角色关系。
- 声音方向（可选，服务有声版本，不阻塞视觉视频链路）。

视频侧必须尽量保证同一角色在不同镜头中的一致性。

### 3. 场景约束

场景约束应至少包含：

- 场景名称。
- 空间类型。
- 时间环境。
- 天气环境。
- 光照方向。
- 色彩基调。
- 关键道具。
- 空间关系说明。

场景约束的目标不是“生成一张好看的图”，而是让多镜头下的地点保持相对稳定。

### 4. 镜头约束

镜头约束应至少包含：

- 景别。
- 机位角度。
- 运镜方式。
- 主体动作。
- 主体情绪。
- 构图倾向。
- 光照要求。
- 禁止项。

镜头约束不能只停留在自然语言 prompt，应该逐步结构化。

当前实现已经开始从“只靠 `visual_prompt`”往“多源约束叠加”演进，真实进入后端 prompt 的输入包括：

- 项目级视觉风格锁定。
- `character_refs`。
- `scene_refs`。
- 已确认的 `shot_first_frame`。

### 5. 音频约束

音频约束应至少包含：

- 对白内容。
- 说话角色。
- 情绪。
- 声音风格。
- 旁白策略。
- 字幕文本。
- 配乐提示。
- 音效提示。

### 6. 禁止项约束

禁止项应贯穿整个视频链路，而不是只在最后一环补一句。

视频侧常见禁止项包括：

- 不要真人感。
- 不要照片级写实。
- 不要角色变脸。
- 不要服装漂移。
- 不要错误手部。
- 不要多余文字。
- 不要水印。
- 不要标志。

## 十、强参考方向的处理原则

用户可能会提出很强的参考要求，例如希望整体效果尽可能接近某一明确作品方向。

这类需求在产品上必须被支持，因为这是非常真实的创作诉求。

但实现原则应是：

- 不在代码中写死某个作品专用逻辑。
- 不为每个作品预先做一套完整模板系统。
- 先把参考对象转化为结构化约束。
- 允许用户在生成前确认这些约束。
- 允许用户在生成后根据差异继续修正这些约束。

也就是说，系统要支持“高度贴近某种参考方向”，但产品实现上应保持通用和可扩展。

## 十一、视频生成主流程

建议视频主流程采用以下结构：

```text
定稿章节
-> 生成分镜
-> 准备角色视觉素材
-> 准备场景素材
-> 生成镜头画面或首帧
-> 生成视频片段
-> 按需生成或修订自动对白脚本
-> 按需生成对白音频、字幕、配乐和音效
-> 合成镜头片段
-> 合成 final.mp4
-> 用户反馈
-> 单镜头或局部重生成
```

这里最关键的是：

- 不是每个项目都必须把所有步骤走满。
- 默认应先走最短闭环。
- 需要更高可控性时，再逐步增加角色素材、场景素材、首帧确认和更精细的音频包装。
- 配音和对白是增强项，不是视觉视频链路的硬前置；视觉效果未达标前，不应让配音成为主流程阻塞点。

当前实现状态已经进入以下形态：

```text
定稿章节
-> 分镜
-> 角色三视图
-> 可选镜头首帧
-> 视频任务
-> 本地 worker
-> 即梦文生视频 / 即梦图生视频-首帧
-> 可选对白、字幕、FFmpeg 合成
-> final.mp4
```

## 十二、产品交互设计

### 1. 主界面原则

主界面应尽量保持单条主流程，而不是拆成太多难以理解的模块。

建议用户看到的主区域为：

- 项目设定。
- 小说生成。
- 分镜生成。
- 视觉资产生成。
- 视频生成。
- 可选音频生成。
- 结果预览。
- 反馈与重生成。

### 2. 默认操作方式

默认应支持一条短路径：

```text
填写项目
-> 生成概要
-> 生成正文
-> 定稿章节
-> 生成分镜
-> 准备视觉并开始视频生产
-> 预览结果
-> 按需补对白、配音、字幕和音效
-> 按意见重生成
```

### 3. 高级控制项

高级控制项不应消失，但应隐藏在按需展开层中，例如：

- 角色视觉锁定。
- 场景参考绑定。
- 镜头结构化编辑。
- 声音角色映射。
- 字幕样式控制。
- 配乐和音效控制。

## 十三、重生成机制

重生成机制是本系统最重要的产品能力之一。

### 1. 重生成层级

系统至少应支持以下层级的重生成：

- 小说概要重生成。
- 单章正文重写。
- 分镜重生成。
- 单镜头重生成。
- 自动对白脚本重生成。
- 单条对白音频重生成。
- 单镜头视频片段重生成。
- 从失败点继续合成最终视频。

当前实现里，已经开始具备“重生成但不乱覆盖”的基础机制，重点不再只是能不能重生成，而是：

- 重生成前是否能识别已锁定内容。
- 预检时是否能跳过已锁定内容。
- 前端是否能让用户明确知道这次跳过了什么。

### 2. 重生成输入

每次重生成都应明确记录：

- 用户不满意的点。
- 本轮需要加强的约束。
- 本轮必须保留的内容。
- 本轮修改范围。

### 3. 重生成保护机制

重生成时默认应保护：

- 已锁定角色外观。
- 已锁定场景。
- 已锁定镜头首帧。
- 已确认对白文本。
- 已确认角色声音。
- 已成功且无需重做的镜头结果。

### 4. 重生成结果说明

系统应尽量让用户看懂：

- 这次改了什么。
- 沿用了什么。
- 为什么这次结果会更接近用户要求。

## 十四、锁定与版本机制

### 1. 锁定机制

用户确认后的关键结果必须可锁定，至少包括：

- 概要。
- 角色设定。
- 角色三视图。
- 场景参考。
- 镜头首帧。
- 对白脚本。
- 角色声音。

当前已落地或已接近落地的锁定点包括：

- 长篇概要 / 章节概要。
- `character_turnaround`。
- `shot_first_frame`。
- `audio_script.audio_script_locked`。
- `media_assets.meta.locked` 下的对白、旁白和其他素材。

### 2. 版本机制

所有关键生成物应保留版本，至少包括：

- 小说概要版本。
- 正文版本。
- 分镜版本。
- 对白脚本版本。
- 音频版本。
- 视频片段版本。
- 最终视频版本。

起步阶段版本体系不必过重，但至少要支持：

- 查看历史结果。
- 恢复明确旧版本。
- 知道当前使用的是哪个版本。

## 十五、工程架构设计

### 1. 分层结构

建议系统分为以下五层：

#### 内容层

负责小说概要、正文、定稿等文字内容。

#### 约束层

负责项目级风格、角色、场景、镜头、音频和禁止项约束。

#### 生成层

负责分镜、对白脚本、配音、视觉素材、视频片段和最终合成。

#### 任务层

负责排队、执行、重试、取消、恢复和状态记录。

#### 反馈层

负责收集用户意见，并将其转为下一轮可执行约束。

### 2. 异步任务

以下任务应采用异步执行：

- 批量正文生成。
- 分镜生成。
- 自动对白脚本生成。
- 批量对白配音。
- 角色视觉素材生成。
- 视频片段生成。
- 最终视频合成。

### 3. 可观察性

每个任务都应至少能看到：

- 当前状态。
- 当前步骤。
- 已完成数量。
- 失败原因。
- 输出结果。
- 最近一次心跳。

### 4. 文件与素材管理

系统必须清楚管理以下资产：

- 角色视觉素材。
- 场景素材。
- 镜头首帧。
- 视频片段。
- 对白音频。
- 旁白音频。
- 字幕文件。
- 配乐。
- 音效。
- 最终视频。

后续应逐步从本地目录过渡到统一存储服务。

## 十六、当前能力与现实判断

结合当前实现情况，系统已经具备一条最小闭环：

- 小说概要生成与修订。
- 正文生成与定稿。
- 分镜生成与编辑。
- 项目级视觉风格锁定。
- 角色三视图生成。
- 镜头首帧生成、预览和锁定。
- 视频任务创建与执行。
- 视频片段生成。
- 最终视频合成。
- 自动对白脚本生成。
- 角色声音配置字段。
- 对白音频生成与试听。
- 对白混入镜头片段。
- 字幕文件生成。

并且当前实现已经新增几个关键的“可控性”能力：

- 视频任务、分镜任务、批量正文任务都开始输出结构化真实进度摘要。
- 本地视频分支会直接复用已锁定首帧。
- 即梦视频分支会在有首帧时优先走图生视频-首帧，否则回退文生视频。
- 预检流程会返回这次“生成了什么 / 跳过了什么”的真实结果，而不是只返回完成状态。
- 镜头编辑保存时不再意外清空 `character_refs / scene_refs`。
- 后端请求、批量正文任务、视频 worker 和前端 API 都开始输出中文日志，降低“跑到哪里了”的排查成本。
- “一键生成小说内容”按钮会直接显示不可用原因，例如概要未锁定、章节范围不合法、章节概要未锁定或已有活跃批量任务。
- “准备视觉并开始视频生产”默认不再强制生成对白脚本和配音，配音保留为用户按需触发的可选链路。
- 三视图和视频提示词已经从通用模板提示增强为项目视觉风格、角色差异锚点、具体镜头构图和反流水线感约束。

这说明系统已经从“概念设计”走到“最小可运行链路”。

但当前更重要的问题是产品化问题，而不是继续扩展零散功能。

## 十七、当前关键缺口

当前关键缺口主要在视频侧：

- 角色一致性还不够稳。
- 场景一致性还不够稳。
- 镜头控制还不够细。
- 当前三视图和视频仍容易出现“流水线感”，需要继续把项目设定转成更可执行的视觉锚点、镜头语言和角色差异约束。
- 字幕、配乐、音效尚未完整进入最终成片；这些属于增强项，不应阻塞视觉样片验收。
- 首帧已进入链路，但更完整的图生视频首尾帧、运镜参数和更强首帧约束还未接齐。
- 视频任务失败后的单镜头恢复能力还不够完整。
- 用户虽然能看到部分状态，但还不够直观。
- 结果虽可生成，但离“持续逼近用户要求”还有距离。

另外还有几个直接影响质量的现实问题：

- `scene_refs` 目前前端仍是最小可用编辑形态，还不是资产选择器。
- 即梦图生视频当前先接首帧模式，首尾帧和运镜模式尚未纳入统一 provider 分支。
- 角色、场景、首帧三类约束虽然已经开始叠加，但还没有形成完全结构化的镜头约束界面。

## 十九、产品完成判断标准

当以下条件基本满足时，系统才可以认为从“工程样机”进入“可持续使用产品”：

- 用户可以顺畅完成从项目设定到最终视频的主流程。
- 小说生成可以支撑后续视频生产，不成为主瓶颈。
- 视频生成能够稳定体现用户指定的主要方向。
- 用户可以对关键结果进行局部修改和重生成。
- 锁定内容不会被后续任务误覆盖。
- 用户可以单独重试失败镜头。
- 用户可以预览和下载最终有声视频。
- 用户不需要依赖后台日志才能理解系统状态。

## 二十、生产链路打通定义

为了避免“代码里有模块”被误判为“链路已经打通”，本项目需要明确区分三种状态：

### 1. 最小链路打通

满足以下条件，才可称为“最小生产链路打通”：

- 可以从项目设定创建一个真实项目。
- 可以生成并锁定长篇概要。
- 可以批量生成正文，并至少完成一章定稿。
- 可以从定稿章节生成分镜。
- 可以生成角色三视图或镜头首帧，并让视频任务真实使用这些视觉约束。
- 可以创建视频任务，并最终产出一个可播放的 `final.mp4`。
- 用户可以在前端看到主要任务状态，而不是只能靠控制台日志判断结果。
- 自动对白脚本、对白音频和字幕属于有声增强链路，不作为最小链路打通的阻塞条件。

### 2. 完整链路打通

满足以下条件，才可称为“完整生产链路打通”：

- 最小链路已稳定通过。
- 锁定的角色三视图、镜头首帧、对白脚本和对白音频会被真实复用。
- 即梦链路和本地 fallback 链路都至少各自完成过一次真实样片。
- 失败后可以定位到具体章节、具体分镜、具体镜头或具体素材步骤。
- 至少支持单镜头级别的重试或重生成，而不是整条任务全部重来。
- 视觉版本中角色、场景、镜头风格基本稳定；有声版本中对白、字幕和画面时间轴基本对齐。

### 3. 产品级打通

满足以下条件，才可称为“产品级打通”：

- 用户可以不依赖开发者手工补库、改状态、改文件，就完成主流程。
- 失败信息、跳过信息、锁定复用信息都能直接在界面中理解。
- 主要生成结果可以在界面内预览、比较、锁定、替换和重试。
- 系统能连续支撑多个项目运行，而不是只在单个样例上偶然成功。

### 4. 当前建议采用的判定口径

近期不要再用“能不能生成”作为判断标准，而应统一用下面的口径验收：

```text
真实项目
-> 真实章节
-> 真实分镜
-> 真实角色三视图
-> 真实镜头首帧或视频提示词
-> 真实视频片段
-> 真实 final.mp4
-> 前端可见状态
-> 失败可定位
-> 锁定内容可复用
```

有声版本在上述视觉链路稳定之后再验收：

```text
真实对白脚本
-> 真实对白音频
-> 字幕/音频进入 final.mp4
-> 可按角色或镜头局部重试
```

只要其中任一关键节点仍需要开发者手工改数据库、手工补素材、手工改任务状态，链路就不应被判定为真正打通。

## 二十一、验收与降级口径

### 1. 环境前置验收

每次判断链路是否打通前，先确认以下前置条件：

- `.env` 中的文本模型、Embedding、MySQL、鉴权配置齐全。
- `frontend/dist` 已构建，后端启动后可直接挂载前端。
- 数据库初始化和手写迁移可自动完成。
- `FFmpeg` 可被后端和 worker 实际调用。
- 若走即梦链路，AK/SK、视频模型 req key、轮询超时配置有效。
- 若走 fallback 链路，图片生成 provider 和 FFmpeg 配置有效；TTS 只在验收有声版本时作为必选前置。

### 2. 主链路验收

主链路验收至少应覆盖以下动作：

1. 创建项目并填写视觉风格约束。
2. 生成长篇概要并提交一次反馈修订。
3. 锁定概要后批量生成正文。
4. 至少将一章正文定稿。
5. 用该章生成分镜，并完成一次镜头编辑。
6. 生成角色三视图，并检查角色差异、服装、配色和项目视觉风格是否符合要求。
7. 生成至少一个镜头首帧并完成锁定。
8. 创建视频任务并拿到最终成片。
9. 在前端确认任务进度、跳过信息和失败信息是可见的。
10. 如需有声版本，再单独生成自动对白脚本、对白音频和字幕，并检查是否进入最终视频。

### 3. 降级链路验收

系统不应只有理想 provider 可用时才能运行，还必须验证降级能力：

- 没有即梦 key 时，是否能自动回退到本地图像 + FFmpeg 分支；有声版本再验证 TTS。
- 没有可用首帧时，是否能从图生视频回退到文生视频。
- 某条对白音频失败时，视觉视频任务不应被阻塞；有声版本应能只重试该条音频，而不是整条视频任务全部失败。
- 某个镜头失败时，是否至少能明确停在镜头级，而不是只得到一个笼统的 `failed`。

### 4. 不算打通的情况

以下情况都不应被记为“链路已打通”：

- 只能通过脚本直接改任务状态来伪造完成。
- 只能依赖开发者手工上传素材补全缺口。
- 只能生成静态素材，不能产出真实 `final.mp4`。
- 失败后看不到明确失败阶段，只能翻后台日志。
- 锁定内容在后续任务里仍然会被普通生成覆盖。

## 二十四、生成前校对与创作上下文包

当前系统在“项目设定”“人物卡”“参考作品”三个入口上已经积累了不少素材，但仍存在一个关键断层：

- 素材是分散填写的。
- 系统如何理解这些素材，对用户不可见。
- 真正喂给小说生成、概要修订、分镜生成、视频生成的内容，对用户不可见。
- 一旦结果出错，用户只能从结果反推问题，排查成本过高。

因此，正式方案需要新增一个主流程阶段：

```text
项目设定 / 人物卡 / 参考作品确认
-> 生成前校对
-> 创作上下文包确认
-> 概要
-> 正文
-> 分镜
-> 视频预检
-> 视频任务
```

这里的“生成前校对”不是附加功能，而应该成为后续所有生成链路共享的上游关口。

### 1. 这个阶段要解决什么问题

它至少要同时解决四类问题：

- 素材冲突检查：
  项目设定、人物卡、参考作品提炼结果之间是否互相矛盾。
- 原著偏差检查：
  如果用户选择“按参考作品内容理解”，系统是否正确识别了原著角色、关系、世界规则和禁区。
- 喂料可视化：
  用户必须能看到系统最终会把哪些内容喂给后续生成模型。
- 上下文冻结：
  一旦用户确认，后续概要、正文、分镜、视频必须共享同一份确认后的上下文，而不是各自随意拼 prompt。

### 2. 生成前校对的输入

这个阶段的输入应至少包括：

- 项目设定
- 世界观设定
- 写作规则
- 人物卡
- 参考作品确认结果
- 长期资料与补充资料

系统不再把这些内容简单视为“提示词原料”，而要把它们统一整理成一份结构化创作事实源。

### 3. 生成前校对的输出

系统应产出一份明确可保存、可复用、可追溯的 `Context Pack`。

建议结构如下：

```text
ContextPack
├─ project_snapshot
├─ character_snapshot
├─ reference_snapshot
├─ source_snapshot
├─ conflict_report
├─ derived_constraints
├─ feed_preview
└─ confirmation_state
```

其中各部分职责如下：

- `project_snapshot`
  项目设定、世界观、写作规则、风格目标等静态快照。
- `character_snapshot`
  角色姓名、性别、年龄感、身份、角色定位、关系、背景、音色设定等统一角色表。
- `reference_snapshot`
  参考作品的结构化理解结果，区分“风格参考”与“内容参考”。
- `source_snapshot`
  长期资料、补充资料、用户附加说明的快照。
- `conflict_report`
  冲突、歧义、缺失项、原著偏差风险。
- `derived_constraints`
  系统从素材中归纳出的硬约束与软约束。
- `feed_preview`
  真正准备喂给模型的结构化上下文预览。
- `confirmation_state`
  用户是否已确认，可用于阻断后续链路。

### 4. 参考作品必须拆成两种模式

这是后续一致性最关键的一点。

#### 4.1 风格参考

适用于用户只是希望借参考作品的：

- 文风
- 节奏
- 情绪密度
- 镜头语言
- 视觉媒介气质

这类模式下，系统只能抽取“可迁移表达约束”，不能默认沿用原著人物和剧情。

#### 4.2 内容参考

适用于用户明确希望系统“理解这部作品本身”，并据此做人物、关系、世界规则、叙事边界上的对照。

这类模式下，系统不能再假设“模型自己知道原著全部内容”。

必须先把参考作品解析成结构化内容，再让用户确认：

- 主要人物是谁
- 角色关系是什么
- 世界规则是什么
- 哪些设定是原著核心
- 哪些地方不能违背原著认知

只有确认后，这部分内容才允许进入后续生成链路。

### 5. 前端必须把“系统理解结果”展示出来

生成前校对页面建议拆成四块：

#### 5.1 原始素材区

展示用户已填写或确认的原始内容：

- 项目设定
- 人物卡
- 参考作品
- 长期资料

#### 5.2 系统整理结果区

展示系统归纳后的统一上下文：

- 核心故事定位
- 主角与关键配角表
- 角色关系摘要
- 世界观规则
- 叙事禁区
- 风格锚点

#### 5.3 风险与冲突区

按优先级展示：

- 必须修复的硬冲突
- 建议修复的弱冲突
- 缺失信息
- 原著偏差提示

#### 5.4 喂料预览区

这里不一定要暴露完整 prompt，但必须明确展示系统真正会喂给后续模型的内容，例如：

- 人物硬约束
- 世界观硬约束
- 参考作品提炼约束
- 可自由发挥范围
- 禁止事项

这样用户看到的不是“系统保存了什么”，而是“系统将如何理解并使用这些内容”。

### 6. 后续所有链路都要绑定同一份 Context Pack

一旦用户确认 `Context Pack`，后续所有关键链路都要显式引用它，而不是各自从项目表、人物卡表、参考作品表零散抓数据。

至少包括：

- 长篇概要生成
- 概要反馈修订
- 正文批量生成
- 正文修订
- 分镜生成
- 视频预检
- 视频任务生成
- 音频与配音生成

这样可以确保：

- 小说链路和视频链路共享同一组人物与世界事实。
- 出现结果偏差时，可以回溯“当时到底喂了什么”。
- 用户修改了人物卡或参考作品后，可以明确提示“当前 Context Pack 已失效，需要重新校对确认”。

### 7. 生成前校对之后，哪些内容属于硬约束

正式方案应把以下内容明确提升为后续生成的硬约束，而不是“尽量参考”：

- 已确认人物姓名
- 性别与身份
- 核心角色关系
- 世界规则与禁区
- 参考作品中被确认保留的内容约束
- 用户明确锁定的叙事边界

而以下内容可以保留为软约束：

- 风格偏好
- 节奏偏好
- 情绪浓度
- 镜头气质
- 语言颗粒度

系统只有先区分硬约束和软约束，后续模型输出才有可能稳定。

### 8. 工程落地建议

后端建议新增：

- `context_packs`
- `context_pack_conflicts`
- `context_pack_feeds`

其中：

- `context_packs` 保存快照和确认状态
- `context_pack_conflicts` 保存冲突与风险报告
- `context_pack_feeds` 保存给不同链路的结构化喂料版本

接口建议新增：

- `POST /api/projects/{id}/context-pack/build`
- `GET /api/projects/{id}/context-pack`
- `POST /api/projects/{id}/context-pack/confirm`
- `POST /api/projects/{id}/context-pack/rebuild`

前端建议新增：

- `生成前校对` 页面
- `喂料预览` 面板
- `上下文失效提醒`

### 9. 与当前问题的直接对应关系

这个阶段引入后，可以直接解决当前暴露出来的几类核心问题：

- 人物卡明明填了，但概要和正文还是换名字。
- 参考作品填了，但系统实际上只拿到了作品名或一小段约束。
- 小说链路和视频链路吃的不是同一份上下文。
- 用户看不到系统到底如何理解自己的素材，只能从错误结果倒查。

因此，生成前校对不是体验优化，而是整个“约束驱动工作台”成立的基础能力。

### 10. 分阶段实施计划

第一阶段：

- 先做 `Context Pack` 数据结构和生成前校对页面。
- 先不要求复杂自动修复，重点解决“看得见”和“能确认”。

第二阶段：

- 让长篇概要生成、概要反馈修订强制绑定 `Context Pack`。
- 把人物、世界观、参考作品约束明确区分成硬约束和软约束。

第三阶段：

- 让正文生成、分镜、视频预检、视频任务全部继承同一份 `Context Pack`。
- 加上上下文失效检测和重新确认流程。

第四阶段：

- 逐步增强自动冲突识别、原著偏差识别、角色一致性校验和生成后回溯展示。

## 二十五、结论

本项目后续的核心，不是把小说和视频两侧都做成同样重的系统，而是明确：

- 小说侧以够用、可修订、可支撑视频为目标。
- 视频侧以可控、一致、可反馈、可重生成为目标。

系统真正要解决的问题，不是“能不能生成”，而是：

- 能不能越来越接近用户想要的效果。
- 能不能在复杂视频约束下仍然保持简单使用。
- 能不能让用户的意见真正改变下一轮结果。

因此，本项目的正式方向可以概括为三点：

1. 小说轻约束，视频重约束。
2. 默认简单，按需增强。
3. 以反馈、锁定和重生成作为核心能力建设方向。
