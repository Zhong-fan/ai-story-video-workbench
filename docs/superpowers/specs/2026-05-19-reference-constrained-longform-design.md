# Reference-Constrained Longform Workflow Design

**Date:** 2026-05-19

**Goal:** 把当前“参考作品 + 项目设定 + 小说生成 + 视频生成”的软提示链路，升级为“硬约束可继承、可校验、可拦截”的长篇创作工作流，同时把前端顶层入口收敛为 `设定 -> 小说 -> 视频` 三阶段。

## Background

当前系统已经具备：

- 项目设定、参考作品解析、人物卡、上下文包
- 长篇概要生成与批量正文生成
- 角色三视图、分镜、首帧、音频、视频任务

当前系统的核心问题不是“能力缺失”，而是“约束失真”和“入口过多”：

1. 项目设定与参考作品主要以自然语言 prompt 存在，系统不能把它们当成硬规则执行。
2. “第 1-10 章不能相遇”这类跨章节规则没有结构化存储，也不会自动分发到每一章。
3. 草稿后处理只检查“该发生的有没有发生”，不检查“禁止发生的有没有被违反”。
4. 视觉链路仍然是“文字 -> 直接生图 -> 继续漂移”，没有参考资产定稿和后续强继承。
5. 前端把大量功能放在同一层导航上，真实主流程被淹没。

## Product Principles

### 1. 项目设定和原作事实都是硬约束

用户写进项目里的规则，以及用户要求默认继承的原作事实，默认都属于硬约束。模型无权自行弱化、折中或忽略。

### 2. 只有授权改写才能覆盖原作事实

系统必须区分：

- 默认继承的原作事实
- 用户明确授权改写的部分

没有授权改写的地方，原作事实保持成立。

### 3. 约束必须可结构化、可继承、可校验

规则不能只存在于自然语言 prompt 中。系统必须能：

- 结构化存储规则
- 自动把系列级规则分发到章节级
- 在概要和正文生成后校验违规
- 对违规结果进行拦截或重写

### 4. 视觉链路必须先定稿再继承

角色三视图不是一次性结果，而是后续镜头与视频的角色母版。未定稿的角色视觉资产不能被当作正式参考。

### 5. 前端以工作流阶段组织，而不是按功能仓库组织

顶层只保留：

- `设定`
- `小说`
- `视频`

其他能力降为阶段内步骤、抽屉或次级入口。

## User Problems to Solve

### 小说侧

- 用户一次性写清 1-10 章的硬边界，但系统仍然偷跑。
- 系统把“必须发生”和“禁止发生”都当成模糊建议。
- 原作改写项目中，系统会静默偏离原作既有事实。

### 视觉侧

- 只写“参考《天气之子》”并不能让角色和视频真正像原作。
- 当前项目没有“自动搜参考图 -> 用户确认 -> 参考资产驱动生成”的正式链路。
- 第一张三视图一旦偏掉，后续视频只会继续放大偏差。

### 前端侧

- 顶层导航入口过多。
- 同一层级混合了主流程和次级工具。
- 用户不能一眼看出当前阶段和下一步。

## Target Workflow

### 1. 设定阶段

用户完成：

- 项目核心设定
- 主要人物卡
- 长篇概要
- 故事边界
- 生成前校对

其中“故事边界”允许用户用自然语言一次性表达跨章节硬约束，例如：

`第 1 到 10 章男女主不要相遇，先分别讲清楚他们相遇之前的故事。第 11 章才第一次正式见面。`

系统必须先将其解析为结构化规则，再交给用户确认。

### 2. 小说阶段

用户围绕当前章节工作，而不是围绕整套后台能力工作：

- 章节列表
- 当前章节概要
- 当前章节生效中的硬约束
- 草稿生成、重写、定稿
- 违规提示与自动重写结果

### 3. 视频阶段

用户完成：

- 参考图确认
- 角色三视图候选生成与锁定
- 分镜生成与镜头调整
- 视频预检
- 视频任务生成

## Core Data Model

### ConstraintRule

统一承载硬约束。

关键字段：

- `source`: `project` / `reference_work` / `authorized_override`
- `scope_type`: `series` / `chapter_range` / `chapter`
- `start_chapter_no`
- `end_chapter_no`
- `rule_type`
- `subjects`
- `predicate`
- `payload`
- `priority`
- `status`

示例：

- 第 1-10 章禁止男女主直接相遇
- 第 11 章允许第一次正式相遇
- 改写起点前的原作关系状态必须成立

### ReferenceFact

承载“默认继承的原作事实”。

关键字段：

- `fact_type`: `character_identity` / `relationship_state` / `world_rule` / `plot_fact` / `visual_target`
- `reference_key`
- `chapter_effective_until`
- `payload`
- `status`

### ReferenceImageAsset

承载自动发现或后续补充的原作参考图。

关键字段：

- `source_work`
- `asset_kind`: `poster` / `character_sheet` / `stills` / `scene_frame`
- `remote_url`
- `provider`
- `source_page`
- `mapped_character_name`
- `status`: `candidate` / `approved` / `rejected`

### CharacterReferenceProfile

承载“项目角色如何继承原作角色”的映射。

关键字段：

- `character_card_id`
- `reference_character_name`
- `visual_reference_asset_ids`
- `locked_turnaround_asset_id`
- `status`: `unmapped` / `mapped` / `turnaround_pending` / `turnaround_locked`

### ViolationCheck

承载概要或正文的违规检查结果。

关键字段：

- `target_type`: `series_plan` / `chapter_outline` / `draft_version`
- `target_id`
- `rule_id`
- `violation_type`
- `severity`
- `message`
- `auto_repair_attempted`
- `resolved`

## Constraint Execution Model

### Planning Phase

在生成长篇概要前：

1. 加载项目硬约束
2. 加载当前有效的原作事实
3. 加载授权改写
4. 将系列级约束分发为章节级有效规则

规划模型必须同时输出：

- 三层概要
- 每章继承到的关键规则快照

### Outline Validation Phase

概要生成后必须做违规检查：

- 是否违反章节区间规则
- 是否违反原作事实
- 是否出现未授权改写

概要违规时不能直接进入锁定状态。

### Draft Generation Phase

每章正文生成时，输入不再只有“本章目标”和“must_not_happen”文字，而是：

- 本章显式硬约束列表
- 本章继承的原作事实
- 本章允许的改写边界

### Draft Validation Phase

正文生成后必须同时运行两种检查：

- `coverage check`: 正向检查，要求发生的是否真的发生
- `violation check`: 反向检查，禁止发生的是否被违反

只要命中硬约束违规，就不能作为成功结果保存。

## Failure Handling

### Constraint Conflicts

系统禁止静默折中。冲突必须明确显示为：

- 原作继承
- 用户授权改写
- 待确认冲突

### Generation Violations

若正文或概要违反硬约束：

- 标记为违规
- 自动尝试重写
- 仍失败则阻止进入下一阶段

### Visual Drift

若角色三视图未锁定：

- 允许生成候选
- 禁止作为正式视频基线

## Non-Goals

本轮不做：

- 一开始就建设全自动高精度视觉相似度评分闭环
- 一开始就特化《天气之子》单作品逻辑
- 把所有现有页面全部重写

## Success Criteria

1. 系列级自然语言规则可以稳定转成结构化硬约束。
2. “第 1-10 章不相遇”类规则可自动分发到章节并被拦截违规。
3. 原作事实默认继承，未授权改写不会静默发生。
4. 角色三视图先定稿，后续视频视觉继承更稳定。
5. 顶层界面只剩三阶段，主流程比当前明显更清晰。
